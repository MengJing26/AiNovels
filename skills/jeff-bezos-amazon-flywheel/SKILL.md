# Skill: 贝索斯亚马逊飞轮引擎

## 概述

将杰夫·贝索斯在亚马逊构建的"飞轮效应"（Flywheel）和"Day 1"操作系统系统化，转化为可持续增长的组织运作框架。

**核心公式**：
```
growth = customer_obsession × operational_excellence × long_term_thinking
```

或者用飞轮表示：
```
飞轮 = (客户体验 → 流量 → 供应商 → 成本 → 价格 → 客户体验...) × Day 1 心态
```

---

## 🔄 飞轮效应的本质

### 定义
飞轮是一个自我强化的正向循环系统。初始推动很费力，但一旦转动起来，惯性会持续加速，无需额外外力。

### 贝索斯的飞轮（经典版本）
```
客户体验 ↑ → 更多流量 ↑ → 更多供应商 → 更低价格 & 更好选择 → 客户体验 ↑
```

每一环节都增强下一环节，形成闭环，而非线性因果。

### Day 1 vs Day 2
- **Day 1**：客户痴迷、逆向工作、快速决策、接受失败、持续创新
- **Day 2**：停滞、流程化、规避风险、远离客户、缓慢反应
- **贝索斯定律**：Day 2 → Day 3 = 死亡。必须永远保持 Day 1 心态

## 🧩 核心模块

### 模块1：客户痴迷诊断器（Customer Obsession Detector）

**功能**：区分"客户痴迷"和"竞争对手 obsessed"，确保决策以客户为中心。

**三问检查**：
1. **我们为谁创造价值？**（客户定义必须具体，非"所有人"）
2. **这个决定客户会在乎吗？**（不在乎 = 内部便利优先）
3. **如果我们做得更好，客户会感知吗？**（不能感知 = 无价值浪费）

**客户价值公式**：
```
customer_value = (功能改善 × 情感增益) / 学习成本
```

- 功能改善：性能/效率提升（+0-10）
- 情感增益：愉悦/信任/安心（+0-10）
- 学习成本：新概念/步骤/困惑（0-∞）
- 值 > 5 才值得做

**贝索斯原则**：
- 竞争对手是"噪音"，客户需求才是信号
- 从客户需求反向推导技术，而非技术推进找场景
- 客户永远"想要更快、更便宜、更方便"，即使他们说不出来

### 模块2：飞轮构建器（Flywheel Builder）

**功能**：设计、绘制、量化你的业务飞轮，识别关键杠杆点。

**步骤**：
1. **绘制飞轮**：列出 4-8 个核心变量，用箭头连接成闭环
   - 例：用户体验 → 口碑传播 → 获客成本降低 → LTV 提升 → 再投资体验
2. **确定方向**：每个箭头的"增强"或"减弱"关系（通常是增强）
3. **量化指标**：给每个变量定义测量指标（e.g., NPS、CAC、LTV、毛利率）
4. **识别薄弱环节**：哪个环节的增强系数最低？哪个阻力最大？
5. **投资薄弱点**：资源优先投入阻力最大的环节（而非最强的）

**飞轮健康度评分**：
```
flywheel_score = (connection_strength × metric_improvement) / friction_coefficient
```

- connection_strength：环节间因果强度（1-10）
- metric_improvement：过去12个月改进幅度（%）
- friction_coefficient：阻力（1-10，高阻力=高摩擦）

> score > 50 健康，< 20 需重构

### 模块3：逆向工作法引擎（Working Backwards Engine）

**功能**：从客户需求出发，逆向定义产品/服务，避免"技术驱动"陷阱。

**六页纸备忘录流程**：
1. **客户新闻稿（PR）**：1页，描述产品发布后客户会怎么说（3年后视角）
2. **常见问题（FAQ）**：3页，客户/销售/内部团队会问什么，如何回答
3. **视觉辅助**：2页，图表/流程图/界面草图
4. **会议评审**：只读文档（无PPT），讨论可行性

**逆向时间线**：
```
T+36个月（客户价值实现） ← T+24（产品上市） ← T+12（MVP完成） ← T+0（决策）
```

每一步必须回答："这个动作如何服务于3年后的客户价值？"

**贝索斯规则**：
- 如果一件事无法写出 PR/FAQ，说明还不够清晰，暂停
- 决策会议前必须完成六页纸，提前发送，会议只讨论内容

### 模块4：Day 1 文化守护系统（Day 1 Guardrails）

**功能**：防止组织滑向 Day 2（官僚、缓慢、脱离客户）。

**Day 1 检查清单**（每周自评 0-10 分）：
| 维度 | 问题 | 得分 |
|------|------|------|
| 客户痴迷 | 我们上周的决策，哪个最直接来自客户反馈？ | |
| 逆向工作 | 我们有多少项目是从PR/FAQ开始，而非技术任务卡？ | |
| 快速决策 | 70%信息充足时决策占比（目标是80%）？ | |
| 接受失败 | 有多少"好失败"（可学习）？有多少"坏失败"（疏忽）？ | |
| 长期投资 | 长期项目（3-5年）资源占比？是否被短期目标挤压？ | |
| 小团队 | "两个披萨团队"（6-10人）占比？大团队协作延迟？ | |
| 外部趋势 | 我们对行业变化的扫描频率？新 competitor 反应速度？ | |

**总分 < 50** → 文化已向 Day 2 滑坡，需干预

**Day 2 症状**：
- 会议变多，决策变慢（"更多对齐"成为借口）
- 用流程替代判断（"按规定办"）
- 关注内部指标而非客户结果
- 规避风险，惩罚失败
- 新想法需要"批准"而不是"实验"

## 🧪 使用示例

### 示例1：SaaS 公司增长飞轮

**现状**：月增长放缓，CAC 升高，客户流失增加

**飞轮构建**：
1. 客户体验（产品易用 + 客户成功）↑
2. 净推荐值（NPS）↑ → 口碑传播 ↑
3. 有机获客（ CAC 降低）↑
4. LTV 提升（留存 + upsell）↑
5. 再投资产品/服务（资源↑）→ 客户体验 ↑

**识别薄弱点**：NPS 只有 20（行业平均 40），说明客户体验不够

**行动**：资源向客户成功团队倾斜，建立"客户痴迷"指标（每周阅读客户反馈）

### 示例2：电商平台 Day 1 检查

**问题**：公司变大后，决策变慢，新项目审批要 3 个月

**Day 1 评分**：
- 客户痴迷：3/10（决策很少来自客户数据）
- 逆向工作：2/10（PR/FAQ 被当作形式）
- 快速决策：4/10（追求 100% 信息）
- 接受失败：1/10（失败导致追责）
- 长期投资：2/10（季度压力挤压长期项目）

**总分 12/50** → 严重 Day 2

**干预**：
1. CEO 每周读客户反馈，分享"最打动/最愤怒"故事
2. 所有项目必须提交 PR/FAQ，会议只读文档
3. "70%信息决策"规则：明确哪些决策可以快速拍板
4. 设立"好失败"奖，季度复盘学习点

### 示例3：逆向工作法应用（新产品）

**传统做法**：
- 技术团队："我们想做 AI 功能"
- 需求讨论：要哪些算法、数据、算力
- 6个月后上线，客户说"我不需要"

**逆向工作法**：
1. **写 PR（新闻稿）**：假设产品已发布，"AI Personal Assistant 帮用户每天节省 2 小时"
2. **写 FAQ**：
   - Q: 保护隐私？
   - A: 本地模型，数据不离开设备
   - Q: 多少钱？
   - A: $9.99/月，但第一年免费
   - Q: 和竞品区别？
   - A: 专注隐私 + 跨 App 自动化
3. **讨论可行性**：需要什么技术、数据、团队？
4. **倒排时间**：T+12个月 MVP → T+6核心算法 → T+0 启动

**结果**：产品更贴近客户需求，避免技术自嗨

## 📚 推荐练习

1. **绘制你的飞轮**：用 Miro/白板画出你公司的增长飞轮，标出每个环节的 KPI，计算飞轮健康度
2. **Day 1 自评**：用检查清单给组织打分，找出 3 个最严重 Day 2 症状，制定 30 天改进计划
3. **逆向工作练习**：选一个正在推进的项目，强制先写 PR/FAQ（1周内），对比传统需求文档
4. **客户痴迷审计**：过去 30 天的决策中，有多少直接来自客户反馈？记录来源（客户、竞品、内部）
5. **薄弱点投资**：识别飞轮中最薄弱的 1 个环节，下周资源倾斜 20% 测试效果
